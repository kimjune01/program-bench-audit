First, thanks for building this. The no-internet, no-install sandbox and the execute-only binary are a real contribution, and I went in trying to take the construction seriously, not to hunt for contamination. My question is a narrow one: does the primary % Resolved metric measure source-blind reconstruction, the way the paper frames it, or does it end up measuring something else? So this is a construct-validity read, not a contamination report.

Here is what I did. I read all 201 released test directories (your 200 tasks plus one empty placeholder) from the public `programbench/ProgramBench-Tests`, and I went through every exact-output assertion, sorting each by how a source-blind, offline solver could actually obtain the value it checks. Three things fell out, and each one comes with a receipt you can re-fetch and check yourself:

- **Recall, at least 21 programs.** Some graded tests want the exact output of a function no source-blind solver can reconstruct from running the binary: non-stdlib hashes (BLAKE3, GOST, xxHash), ciphers (age), compressors (zstd, brotli, lz4), codecs (LPC10, WebP/libjpeg in C), opaque binary formats (BAM, Parquet, Avro, ELF), embeddings (fastText), the Unicode width table. The only offline route to that value is already holding the spec, which is exactly what the no-internet rule forbids and the prompt tries to suppress. Since % Resolved is conjunctive, one of these forecloses the whole task.
- **Oracle provenance, at least 29 programs.** A lot of the graders write their own golden from the reference run (`if not golden.exists(): golden.write_text(result.stdout)` then `assert result.stdout == golden.read_text()`). The oracle is then the reference's own bytes, so the test is really checking byte-identity-to-reference, not a contract. Twelve use a conditional that would pass any output if the golden were ever stripped; I checked, and the goldens ship, so that is a latent risk rather than a live one. This is the classic test oracle problem (Barr et al. 2015; Weyuker 1982).
- **Coverage.** Even setting recall aside, the hidden, one-shot, conjunctive setup makes the solver's real target the whole behavioral surface, not the finite suite. And that surface is measurable from your own tests: a median of 204 distinct graded exact-output obligations per program, up to 4,064 for gdal. A zero % Resolved floor falls right out of a surface that size under a conjunction, with no missing-algorithm story needed.

For anyone running the benchmark, I put together a **model-blind subset**: a 24-program skip-list (no source-blind solver resolves these) and a 171-program benchable subset to report % Resolved over. I computed the list from the test bodies, never from any model's results, so adopting it carries no conflict of interest, and every verdict is re-derivable by grep over the public suites without trusting me.

Repo with the database, per-program receipts, and the re-runnable pipeline: https://github.com/kimjune01/program-bench-audit. Companion write-up: https://june.kim/programbench-measures-recall

**Right of reply.** If the team disputes any specific verdict, I will incorporate corrections and cut a new versioned release. The mechanical spine (the literal-match inventory and the witness retrieval commands) is re-derivable by grep, so corrections can be checked against the suites rather than against my word.
